Confidential — do not distribute

01  North Star
Palate Map is a taste intelligence platform. It starts with film and expands across all creative domains — restaurants, music, books, travel. The core promise is not "what to watch" but "who you are as a viewer, and what that means for every aesthetic choice you make."
"Letterboxd tells you what you watched. Palate Map tells you who you are."
The product is opinionated, precise, and personal. Every score is user-authored. The system reveals patterns in what you've already decided, rather than deciding for you. It should feel like something you'd want to send to friends — a taste identity, not a database.

Tagline: "Taste is everything."
Brand voice: "Everyone has a palate." Taste is not gatekept — it's cultivated. The product name grounds everything: mapping your palate across film (and eventually all creative domains).

Brand Voice
• Tone: like a formative teacher. Socratic, never prescriptive. Observational, not judgmental.
• Core belief: "Everyone has a palate." Taste is not gatekept — it's cultivated.
• 2nd person, present tense. Talks to the user about themselves, not about film in the abstract.
• Dislikes: the word "archetype" (too clinical). Prefer "palate type" or just lean into the name.
• Never tells the user what to think. Reveals what they already think.

Design Principles
• Swiss-editorial. Typographic-first. Feels like a tool a film critic would actually use.
• High fashion / editorial energy — the platform should let your taste shine, not compete with it.
• Dark surfaces for identity moments (archetype reveal in onboarding, palate block in profile).
• No dark mode yet. Clean paper aesthetic is intentional and differentiating.
• Animations: subtle fadeIn only. No bounces, no flashy transitions.
• Mobile: responsive bottom bar redesigned. Tab strip (5 items: Rankings, Analysis, Friends, Watch, Profile) sits above a full-width persimmon "Search" CTA bar (opens global search overlay). The .mobile-bottom wrapper handles fixed positioning; ::after pseudo-element matches action color to cover browser chrome gap transitions. Screen padding accounts for both rows.
• Rankings nav icon is a medal (ribbon + circle), not a hamburger. Analysis uses bar chart. Icons are distinct across all 5 tabs.

04  React: The Decision
Start with Vite + vanilla JS, migrate to React in Phase 3 when the friends layer lands.
Why not React now: The app is data-rendering with light interactivity. Vanilla JS handles it fine.
Why we'll want React later: Friends layer introduces real-time updates. At that point, managing UI state manually becomes painful.
Decision: Build Phase 0–2 in vanilla JS. Reassess at Phase 3 kickoff.

Supabase tables: palatemap_users, palatemap_friendships

palatemap_users

Watchlist Object Schema
tmdbId              number  TMDB film id
title               string

poster              string  TMDB path e.g. "/abc123.jpg"
director            string  (may be empty for entity-stub sourced items)

palatemap_friendships
requester_id        uuid → palatemap_users.id
addressee_id        uuid → palatemap_users.id
status              text  'pending' | 'accepted'
(unique constraint on requester_id, addressee_id)

Film Object Schema
title               string
year                number
total               number (recalculated on weight change — never trust cached)
director            string comma-separated
writer              string comma-separated
cast                string comma-separated (top 8 from TMDB)
productionCompanies string comma-separated
poster              string  TMDB path e.g. "/abc123.jpg"
overview            string  synopsis from TMDB
scores              object  { plot, execution, acting, production,
                             enjoyability, rewatchability, ending, uniqueness }

Scoring Formula
total = Σ(score[cat] × weight[cat]) / Σ(weight[cat])
Weights are user-specific. recalcAllTotals() must be called after any weight change.
Score scale: 0–100 (not 0–10).

9 Archetypes (internally) / Palate Types (brand voice)
Visceralist · Formalist · Narrativist · Humanist · Completionist · Sensualist · Revisionist · Absolutist · Atmospherist
Each has: weight distribution, description (2–3 sentences, 2nd person, observational), quote.
Derived from 6-question onboarding quiz. Editable post-onboarding via weight editor (click username in masthead → Profile → "Edit weights →").

06  Screens & Features (Current)
Cold Landing
Full-screen overlay for new/logged-out users. Layout: centered hero ("Taste is everything." + "Everyone has a palate. Palate Map maps yours."), then a rule, then a 50/50 grid (responsive, stacks under 760px).
Left column: 4 numbered value props with blue number markers (01–04): "Your formula. Not an algorithm." / "Know if you'll love it before you watch." / "Rate what you've seen. Discover what you are." / "Taste is more interesting next to someone else's." — each with a 2-line description, no jargon, new-user-friendly.
Right column: dark surface card ("Your taste has a shape. Let's find it." + brand copy), Google sign-in button, "or" divider, "Continue with email →" (persimmon), Log in / Import links in readable on-dark contrast. Card stretches to match left column height. Footer: "Starting with film. Expanding to everything."

Onboarding (3 entry points)
New user: cold landing → auth step (Google OAuth primary, or name + email → magic link) → 6-question quiz (dark surface intro block: "Six questions. Your taste, revealed.") → archetype reveal (dark surface: name + description + quote + secondary + username) → "Enter Palate Map →" → welcome modal (one-time, localStorage flag).
Returning user: "Log in →" link on auth step. Shows Google OAuth or email-only magic link (no name field). On click: redirects to Google or sends magic link. Supabase session auto-loads profile on return. Legacy profiles with no auth_id are auto-linked on first sign-in (auth session linked to localStorage user, auth_id + email written to Supabase row).
Letterboxd import: drag-and-drop ratings.csv → parses star ratings (×20 = score) → loads films → proceeds to quiz. CTA on name step ("On Letterboxd? Import your ratings →").
Quiz overlay: flex-column, flex-start, 48px top padding so content is scrollable and never clips. "← Start over" button renders below the card (outside it) when a pending auth session exists — plain border style, visually separated, hidden on all non-quiz steps.
Welcome modal: fires once after obFinish() (palatemap_welcome_shown localStorage flag). Shows name + archetype type + 3 first steps (rate 10 films / try Predict / invite a friend). "Let's go →" dismisses.

Rankings (Main Screen)
Default: poster grid (5 columns). Sort pills above grid control sort order + which score badge shows on each card. Toggle to table view (8 category columns + total).
Score badges overlay bottom-right of each poster: color-coded by tier (gold/green/olive/amber/dim).
Rank number overlays top-left of each poster.
Click any card/row to open film modal.
Empty state: 60vh centered CTA — "Start with one you love." + persimmon "Rate your first film →" button.

Film Modal
TMDB poster · synopsis · rank #X of N · clickable director/writer/cast/company chips (→ Explore) · per-category scores + rank + weight bars · inline score editing (Edit scores → sliders → Save) · nearby films in rankings (leaderboard style with rank, diff indicator, current film highlighted in ink).
Chip portraits: director/writer/cast chips show 20px TMDB circle portraits; company chips show 18px rounded-square logo — both load async after modal opens.
If the film was predicted before rating, the modal shows the predicted total vs actual total with delta and accuracy label ("Nailed it" / "Close" / "Off the mark").
Poster correction: view mode includes a "Fix poster" action. Selecting a new TMDB match updates poster, overview, TMDB id, and all linked metadata (director, writer, cast, production companies) in-place, then saves and syncs. If the user picks the wrong replacement, they must correct it again manually.
Film removal: view mode includes "Remove from list" with a confirm-before-delete step. Removes from rankings, updates state, syncs to Supabase.
Friends context: after the breakdown, a "Friends" section shows any friends who have rated the film (their score + rank in their list) or have it on their watch list. Loaded from window._friendsDataCache — background-populated on app init via a single Supabase query for all friends' movies + watchlists.

Explore
Index tabs: Directors / Writers / Actors / Companies / Years. Entities with 2+ films, sorted by avg. Each row shows rank + TMDB portrait (40px circle for persons, 40px rounded-square contained logo for companies) + name + film count + score badge. Images load async.
Entity page: avg score, rank among peers, film count, TMDB portrait in dark header, per-category breakdown (Craft / Experience axis groups) with per-category ranks, AI insight blurb, full film list with poster thumbnails and score columns.
Empty state: "Terra incognita." dashed-border uncharted treatment.

Predict
Search unrated films. Builds taste fingerprint. Finds comparables by shared director/cast. Calls Claude via Cloudflare proxy. Returns predicted score per category, confidence, 2nd-person reasoning anchored to user's specific films. Results cached in currentUser.predictions (keyed by tmdbId) and synced to Supabase — subsequent opens show cached result instantly with "Re-predict →" available. ＋ Watchlist and Rate now → actions on result.
"Find me a film →": button below search input. Fetches TMDB popular (2 pages) + 1 genre-seeded discover page using user's most-rated genre. Filters out rated + watchlisted films. Takes top 6 by quality-weighted vote_average. Runs 6 Claude predictions in parallel. Shows top 3 ranked by predicted total as recommendation cards. Each card: poster, title, predicted total, confidence, truncated reasoning, "+ Watch List" and "Full prediction →" buttons. "Full prediction →" loads cached result (no second API call). "Refresh →" advances page offset for fresh candidates.

Analysis
Editorial header. Category averages split into Craft (Plot/Execution/Acting/Production) and Experience (Enjoyability/Rewatchability/Ending/Uniqueness) axis groups, shown as horizontal score bars. Below: Explore tabs — Directors / Writers / Actors / Production Co. / Years — ranked entity lists with click-through to entity detail pages.

Calibrate
Elo-based head-to-head. Choose category + intensity (15/30/50 matchups). Review proposed changes. Apply writes scores + recalcs + syncs.
Accessible via Profile screen "Calibrate your scores →" button (not in bottom nav).

Add Film
4-step: TMDB search → cast/crew/company curation → 8-category scoring → result card.
Score guide in step 2 reflects actual LABELS thresholds (90/85/80/75/70/65/60/55/50) from state.js.
Poster correction: curation step includes a "Wrong poster? Choose another match →" action. Opens alternate TMDB search results; selecting a replacement swaps in the new poster, metadata (title, year, director, writer, cast, companies, overview), and TMDB id before proceeding to scoring.
Predicted vs actual: if the film was previously predicted, the Step 4 confirmation card shows predicted total vs actual total with delta and a qualitative accuracy label ("Nailed it" ≤3, "Close" ≤8, "Off the mark" >8). Delta is stored in the prediction cache entry (predictedTotal, actualTotal, delta, ratedAt).

Nav Strip
Top bar above nav tabs. Contains: "Add a film" (primary) + "Invite a friend →" (secondary). "Invite a friend →" opens invite modal (not nav to Friends screen).

Two tabs: Friends | Outgoing (with count).

On accept: friendship row written with status='accepted'. Token-based invite links bypass pending — show confirmation overlay ("Add [Name] →" / "Dismiss") then write friendship directly.

Friend profile page
Dark header: friend's archetype in their palette color, display name, username.
Three tabs: Rankings | Taste | Overlap.

Rankings tab (default): paginated list of friend's films sorted by score (10 at a time, Load more). Each row: poster thumbnail, rank, title, year/director, score in friend's color. Click any row → rich film modal with dark header (poster, friend's rank/score in their palette color), then: if user has rated it shows their own score + "View →" to their modal; if unrated shows ＋ Watchlist + Rate now → CTAs; always shows friend's per-category breakdown bars below.

Taste tab: solo radar (friend's archetype weights normalized by their max weight — same shape as their polygon in the overlap view) + category averages bar chart (Craft / Experience groups) + top-5 entities per type (directors/actors/writers/companies, min 2 films, sorted by avg score). Entities show TMDB portraits/logos loaded async. Each entity is expandable (▾/▴) to show inline film list. Film rows are clickable → score modal.

Overlap tab: compatibility score (cosine similarity on weights + co-rated film agreement), dual radar overlay (both weight vectors in respective palette colors), co-rated films list sorted by biggest disagreement, AI-generated compatibility blurb via Claude proxy.
Shared Watch List: if both users have watchlisted the same film, it surfaces in the Overlap tab as "Both want to watch."
Overlap Predict: search bar at the bottom of the Overlap tab. Select any unrated film → sends both taste profiles (films, weights, archetypes) to Claude → returns a combined predicted score + reasoning that distinguishes what each person would respond to by name, anchored to their actual rated films. ＋ Watchlist button on result.

07  Known Issues & Tech Debt
• No test coverage. Shipping by feel.
• Cloudflare Worker renamed to palate-map-proxy — URL updated in all modules (predict.js, friends.js, insights.js).
• Refreshing while inside a friend's profile page lands on friends list (friend ID not persisted). Acceptable for now.
• Entity stub "Rate →" button navigates to Add Film screen and pre-fills search — relies on DOM timing (setTimeout 100ms) which is fragile. Revisit if unreliable.
• Watch list director field is empty for items added from entity stubs (TMDB discovery results don't include director). Acceptable for now — the title pre-fill covers the core use case.
• Auth quiz loop: mitigated with auto auth_id linking (legacy profiles now link on first sign-in) and "← Wrong account? Sign out" escape hatch on quiz intro when a pending auth session is detected. Root cause (no localStorage on first device) is expected — escape hatch is the UX solution.
• Friends data cache (window._friendsDataCache) is loaded on init and when Friends screen is visited. If a friend adds a new film between sessions, the modal context will be stale until next app load. Acceptable for now.
• Prediction cache (currentUser.predictions) is per-user, client-side. No server-side invalidation. Re-predict available manually.
• TMDB search can return an incorrect first-match poster/record for some titles. Users can now correct these manually during Add Film (curation step) and from the ranked-film modal ("Fix poster"). If a user picks the wrong replacement, they must correct it again manually.

12  Ownable Intelligence — Strategic Principle

Core principle: every engineering decision should increase the intelligence Palate Map owns outright and decrease reliance on any third-party API or service. The goal is not to eliminate the Anthropic API dependency immediately — it is to ensure that dependency becomes increasingly substitutable over time, and that the irreplaceable value lives in Palate Map's data and architecture, not in any external model.

What Palate Map owns
• User rating data — every score, weight adjustment, calibration session, and prediction delta. This compounds with time and has no equivalent at any competitor.
• The taste model architecture — the 8-category scoring system, archetype framework, weighting engine, and Elo calibration logic. These are Palate Map inventions, not API outputs.
• The prediction reconciliation loop — predicted vs actual deltas, per-category bias vectors, confidence calibration data. This is the seed of a proprietary learning system.
• The social graph — friendship connections, overlap scores, and compatibility data. A network asset with no external equivalent.
• The prompt architecture — structured so the underlying model (Claude, GPT, or a future fine-tune) can be swapped without changing the surrounding system.

What is currently rented
• Language model reasoning — the Anthropic API performs the final prediction inference. If Anthropic changes pricing, terms, or availability, predictions are affected. This is the primary substitution risk.

Engineering rules that follow from this principle
1. Never put scoring logic, archetype detection, or weighting calculations inside a prompt. These must live in code (state.js, predict.js) and be passed as structured data to the model, not derived by it.
2. Every prediction call must produce structured, storable output — scores, confidence, reasoning, and delta when rated. No intelligence should be generated and discarded.
3. The prompt interface must be model-agnostic. System prompts and user prompts should not reference Claude or Anthropic by name internally, and should be written so a different model could be substituted by changing one import or config value.
4. As prediction volume grows, the reconciled prediction dataset (predicted scores + actual scores + deltas) is a first-class asset to be preserved and structured for eventual fine-tuning use.
5. Bias correction vectors, confidence calibration data, and per-category accuracy metrics should be stored on the user object and updated over time — not recomputed from scratch on each call.
6. Avoid using any external service for logic that could reasonably be computed locally. Examples: archetype detection (cosine similarity, already in JS), compatibility scoring (already in JS), total score calculation (already in JS). These must never migrate into API calls.

The substitution path
Phase 1 (current): Claude Sonnet via Anthropic API. Full reasoning capability, higher cost, external dependency.
Phase 2 (future, ~1,000+ paid users): Evaluate fine-tuned smaller model (e.g. a quantized open-source model hosted on Modal or Replicate) for auto-predict calls. Reserve Sonnet for manual predictions and deep-dive features. Target: cut API cost by 60–80% while maintaining accuracy on routine predictions.
Phase 3 (future, ~10,000+ paid users): The reconciled prediction dataset is large enough to fine-tune a dedicated Palate Map prediction model. This model is trained on real user prediction errors, owns the task completely, and runs at a fraction of the per-call cost. This is Direction 5 of the prediction learning roadmap.

IP protection posture
• Trademark: file "Palate Map" with USPTO (one-time ~$250–350, recommended before public launch).
• Trade secrets: keep prompt architecture, scoring formula, and prediction logic out of client-side code. The Cloudflare Worker proxy already ensures prompts are not visible in network requests. Maintain this.
• Terms of Service: include anti-scraping, anti-reverse-engineering, and prohibition on using Palate Map data to train competing models.
• Copyright: code and UI content are automatically protected. No registration required, though registration strengthens legal standing.
• Real moat: user data depth, social graph lock-in, and execution speed. Legal protection is secondary to these.

Note for Claude Code sessions: when implementing any feature that involves intelligence, prediction, scoring, or user modeling, default to keeping the logic in JavaScript rather than delegating it to the API. Only use the API for natural language generation tasks (reasoning text, blurbs, explanations) that genuinely require it. If a feature can be computed deterministically from user data, compute it in code.
